# 04 - Authentication & Security Architecture

## 1. Core Security Philosophy

DoConnect AI employs a strictly stateless, token-based authentication model. There are no server-side HTTP sessions (e.g., `JSESSIONID`). This decision ensures that both the Main API (Port 8080) and the Chat Service (Port 8090) can scale horizontally without requiring sticky sessions or a distributed session cache like Redis.

## 2. JWT Implementation

Authentication relies on JSON Web Tokens (JWT) signed with the `HMAC-SHA256` (HS256) algorithm. 

### 2.1 Token Generation (`JwtService`)
When a user logs in via `/api/auth/login`, the `AuthService` verifies their BCrypt-hashed password. Upon success, the `JwtService` generates a token containing the following custom payload (claims):
*   `sub`: The user's email address (Subject).
*   `userId`: The database primary key of the user.
*   `name`: The user's display name.
*   `role`: The user's RBAC role (`USER`, `MODERATOR`, `ADMIN`).
*   `iat`: Issued-at timestamp.
*   `exp`: Expiration timestamp (configurable via `app.jwt.expiration-minutes`).

*Why include so much in the payload?* Embedding the user ID, name, and role directly into the token allows downstream services (like the Chat Service) to authenticate and identify the user *without* needing to make an expensive database trip to the `users` table on every request.

### 2.2 API Request Interception (`JwtAuthenticationFilter`)
Every incoming HTTP request to protected endpoints is intercepted by `JwtAuthenticationFilter` (extending `OncePerRequestFilter`).
1.  Extracts the `Authorization: Bearer <token>` header.
2.  Validates the HMAC signature using the shared secret (`app.jwt.secret`).
3.  Checks expiration.
4.  If valid, it constructs an `AppUserDetails` object and populates the Spring `SecurityContextHolder`.

## 3. WebSocket Security

Securing persistent WebSockets requires a different approach than standard HTTP, as headers cannot easily be attached to the initial browser `WebSocket` handshake.

### 3.1 The STOMP Connect Frame
Instead of securing the HTTP upgrade request, security is enforced at the STOMP protocol level.
*   The frontend establishes the raw WebSocket connection.
*   The frontend sends a STOMP `CONNECT` frame containing the `Authorization` header.
*   The Spring `WebSocketAuthChannelInterceptor` intercepts the `CONNECT` frame.
*   It extracts the Bearer token, validates it using the exact same `JwtService` logic, and assigns the user principal to the session accessor. If the token is invalid or missing, an `AccessDeniedException` is thrown, terminating the STOMP connection.

## 4. Role-Based Access Control (RBAC)

Spring Security's `@EnableMethodSecurity` allows granular endpoint protection using annotations.
*   `@PreAuthorize("hasAnyRole('MODERATOR', 'ADMIN')")`: Secures the moderation dashboards.
*   `@PreAuthorize("#authorId == authentication.principal.id or hasRole('ADMIN')")`: Ensures that only the creator of a question (or an admin) can mark an answer as "Accepted."

## 5. Inter-Service Security (The Internal Bridge)

A critical vulnerability in microservice architectures is exposing internal communication endpoints to the public internet. The Chat Service must tell the Main API when a new message is sent so a notification can be pushed.

**The Solution:**
1.  The endpoint `/internal/notifications/chat-message` is configured in `SecurityConfig` to `permitAll()`.
2.  However, an `InternalNotificationFilter` (or Controller-level validation) strictly requires an `X-Internal-Token` header.
3.  This token is a cryptographically secure 256-bit string shared between the two microservices via `.env` variables (`NOTIFICATION_INTERNAL_TOKEN`).
4.  If an external actor attempts to hit `/internal/notifications/*` without the exact matching token, the request is rejected with a `403 Forbidden`.

## 6. Failure Scenarios & Tradeoffs

*   **Tradeoff: Token Revocation**
    *   Because JWTs are stateless, they cannot be individually revoked before their expiration time without introducing a database-backed "Token Blacklist."
    *   *Decision:* To maintain high performance, DoConnect AI does not use a blacklist. Instead, token expiration times are kept relatively short (e.g., 60 minutes). If an admin bans a user, their current session remains active until the token expires.
*   **Security Risk: Shared Secret Management**
    *   Both the Main API and Chat Service *must* have the exact same `app.jwt.secret` configured. If they drift, the Chat Service will reject valid tokens generated by the Main API.
